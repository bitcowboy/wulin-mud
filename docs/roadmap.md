# 路线图 · Roadmap

> 三阶段推进，对应原始设计文档的"响应 → 联动 → 演化"。每个阶段有明确的"验收闸"，没通过不进入下一阶段。

## 阶段总览

```
v0.1 · 响应（Response）       │  2-3 weeks  │  NPC 持续性
v0.2 · 联动（Causation）      │  4-6 weeks  │  局部因果传导
v0.3 · 痕迹（Sedimentation）  │  3-4 weeks  │  玩家行为沉淀
v1.0 · 演化（Evolution）      │  open       │  长期世界历史
```

---

## v0.1 · NPC 持续性

**目标**：让玩家和同一个 NPC 反复交互 10 次以上后，依然能感受到这是一个有连贯记忆、稳定人设、合理情绪演化的"真实存在的人"。

### 必须做

- [ ] Ontology 核心对象（NPC、Memory、Relationship、Location、Item）的 schema + SQLite 持久化
- [ ] Action Type 框架：validate / execute / declare_side_effects 标准接口
- [ ] 15 个基础 Action（Greet、Talk、AskAbout、BuyItem、GiftItem、OffendNPC、Observe、MoveTo、Eavesdrop、SpreadRumor、HelpNPC、Threaten、Wait、Leave、Lie）
- [ ] LLM 调用层与 prompt 模板，对接 OpenAI API
- [ ] Memory 生成 + 衰减机制
- [ ] World tick 调度器（最小版，主要驱动 Memory 衰减和 mood drift）
- [ ] CLI 交互：玩家自然语言输入 → 意图识别 → Action 路由 → LLM 响应
- [ ] 孙婆婆完整种子数据 + 至少 5 个其他核心 NPC 种子
- [ ] 6-8 个核心地点的基础数据
- [ ] 10 个孙婆婆专属测试场景 + eval runner

### 不做（明确推后）

- 战斗系统（v0.2）
- 派系联动（v0.2）
- 玩家行为对宏观世界的影响（v0.3）
- 镇外世界
- 玩家自定义角色

### 验收闸

**唯一的成功标准**：

> 一个独立测试者（没参与开发）与孙婆婆自由交互 30 分钟以上，结束后被问"你觉得她是个什么样的人"，能给出连贯、具体、与种子设定相符的描述；并且能说出至少 3 件他和她之间"发生过的事"。

达不到这一标准前，不进入 v0.2。

### 第一周里程碑（早期里程碑细分）

- **Day 1-3**：仓库骨架、Ontology 核心 schema、SQLite 持久化跑通
- **Day 4-7**：Action 框架 + 3-5 个基础 Action + 孙婆婆 NPC 种子加载
- **Day 8-12**：LLM 集成 + 对话生成 + Memory 写入与检索
- **Day 13-17**：World tick + Memory 衰减 + mood drift
- **Day 18-21**：CLI 完善 + 5 个其他 NPC + 内部 dogfooding
- **后续**：补全 NPC、跑 eval、调参

## v0.2 · 局部因果传导

**目标**：让玩家理解"世界中的变化不是孤立发生的"。一件事的发生，会在小范围内引发其他人和系统的连锁反应。

### 必须做

- [ ] Faction Object Type + 三股暗中势力的派系数据
- [ ] Rumor 传播机制：基于 NPC 关系网的扩散 + 信息扭曲
- [ ] SupplyLine / Business / Reputation 等中观状态对象
- [ ] Propagation Rule 框架：某个 Action 触发后，跨对象的状态变化如何级联
- [ ] 第一条完整因果链：押镖任务失败 → 镖局威信下降 → 镖局生意下滑 → 关联 NPC 情绪变化 → 镇上谣言扩散
- [ ] 战斗系统（最简版）
- [ ] NPC 自主行为：world tick 时让 NPC 根据 goals 选择 Action

### 验收闸

> 一个测试者主动做出"破坏性行为"（让某次押镖失败、得罪某个有关系网的 NPC、传播一条恶性谣言），三天游戏内时间后，必须能在镇上至少 3 个其他 NPC 那里观察到该行为的延伸后果，且这些后果在叙事上自洽。

---

## v0.3 · 玩家痕迹沉淀

**目标**：让玩家感受到"我做过的事真的留在了这个世界里"。

### 必须做

- [ ] 玩家行为审计日志的玩家可见层：可以"翻看"自己做过的关键事
- [ ] NPC 群体的"集体印象"机制：镇上对玩家的整体看法（不只是单个 NPC）
- [ ] 玩家声望系统：在不同派系、不同人群中的差异化声望
- [ ] 长时跨度的 Memory 压缩与"传说化"：玩家很久以前做的事会被传成版本走样的故事
- [ ] NPC 之间互相讨论玩家的机制（在 world tick 中触发，但 token 预算控制好）

### 验收闸

> 测试者玩满 5 个游戏日后离线一周，重新回来时，至少有 3 个 NPC 会主动以与他历史行为相关的方式开场（不是固定开场白），至少 1 个新出现的过客 NPC 会以"听说过"的方式提到他。

---

## v1.0 · 长期演化

**目标**：从"局部联动"走向"整体世界成立"。世界开始有自己的历史轨迹、文化沉淀、可传播的故事。

### 探索方向

- 多人同服（玩家行为相互影响）
- 镇外世界开放（其他小镇、官道、野外）
- 派系长期博弈的可见层
- 季节/节日/灾害等宏观周期
- 玩家创建的传说被官方提炼为"赛季记忆点"
- 数据反哺：用游戏内行为数据持续微调 LLM 的 NPC 表达层

**这一阶段没有明确的"完成"，是持续运营。**

---

## 技术债与重构节点

预计在以下节点需要重构：

- **v0.2 启动时**：SQLite → PostgreSQL + pgvector（为了 Memory 语义检索）
- **v0.3 启动时**：单进程 → 后台 worker（Memory 衰减和 NPC 决策异步化）
- **v1.0 启动时**：从单机 CLI 改造成 client-server，准备多人

## 风险登记

| 风险 | 影响 | 缓解 |
|---|---|---|
| LLM 漂移控制不住 | v0.1 失败 | 用 eval 框架持续回归，把 prompt 模板和约束系统当一等公民对待 |
| Token 成本失控 | 项目难以维持测试 | 上文中的优化（Memory top-N、context 截断、cache） |
| Memory 衰减参数难调 | NPC 显得健忘或偏执 | 在 eval 中加入"长期一致性"场景 |
| Action 设计不当导致世界僵硬 | 玩家感受不到自由 | 保持 Action 列表的克制（先少后多），用户测试驱动新增 |
| 单 NPC 验证成功 ≠ 多 NPC 成功 | v0.1 通过但 v0.2 翻车 | 在 v0.1 后期至少跑通 5 个 NPC 的横向一致性测试 |
